# Architectural Decision Record

Immutable record translated from the original Portuguese in [[raw/context-md-2026-06-11.pt.md]]. Living decisions have their own page linked below.

| Decision | Rationale |
|---|---|
| PWA instead of native app | Single codebase, no App Store, instant updates |
| SQLite initially | Simplicity. PostgreSQL when there are multiple users |
| Pydantic v2 | Modern standard, native FastAPI integration |
| FastAPI | Automatic docs, performance, async support |
| Next.js + Leaflet | Familiarity + Leaflet is free and robust |
| Event-Driven in Camada 3 | Decouples ingestion from consumers |
| Adapter pattern in Camada 4 | Add sources without touching the rest |
| Pure function `montar_alerta()` | Separates mapping (testable) from I/O (network + database) |
| Dedup by `cod_alerta` (PK) | Integrity guarantee at database level |
| Retry only on 5xx/timeout/connection | 4xx errors don't resolve with retry; 408/429 are exceptions |
| Exponential backoff (2s, 4s, 8s) | Industry standard — respects downed API without hostility |
| BackgroundScheduler instead of Blocking | Enables clean `Ctrl+C` shutdown on Windows; prepares future FastAPI integration |
| `time.sleep(1)` on main thread | Works on any OS, respects `Ctrl+C` natively |
| `max_instances=1` + `coalesce=True` | Prevents execution pile-up if a round takes longer than the interval |
| Mock `time.sleep` in tests | Full suite in < 1 second |
| src layout (`src/alertavida/`) | Python Packaging Authority recommended — forces correct imports, avoids relative import pitfalls |
| `pyproject.toml` instead of `requirements.txt` | Modern standard (PEP 517/518); centralizes deps, build, tool config |
| Database in `data/` | Separates runtime-generated data from source code |
| `pip install -e` (editable mode) | Code changes appear instantly without reinstalling |
| Coordinates as optional Value Object (`Coordenadas \| None`) | Data honesty — source does not always provide with precision |
| Municipality always required on Alerta | Minimum geographic fallback guaranteed |
| Enrichment (IBGE, lookups) only in Camada 4 | Camada 2 represents, Camada 4 enriches |
| AlertaResolvido by inference (N=3 absent rounds) | CEMADEN status is always 1; alerts vanish from feed without notice; N=3 is conservative (15 min) and configurable |
| Outbox Pattern in SQLite | INSERT of alert and event in same transaction — eliminates dual-write; natural path to Postgres LISTEN/NOTIFY and later broker |
| ChangeDetector as pure function | Zero I/O; testable without database mock; separates decision (detector) from execution (database) |
| EventBus in-memory without library | ~50 lines of own code; no external coupling; replaceable by broker when needed |
| `ult_atualizacao` as AlertaAtualizado trigger | Explicit field delivered by CEMADEN; more reliable than field hash |
| `codibge` parsed in `from_dict` | Field present in payload; avoids external lookup planned for Camada 4 |
| OutboxDispatcher every 30s in scheduler | Acceptable latency for Camada 3; Camada 7 may reduce if notifications demand |
| uv instead of direct pip | Cross-platform lockfile, guaranteed reproducibility across machines and CI |
| CI matrix Ubuntu + Windows | Paths and encoding behave differently between OSes — platform bugs surface before users find them |
| Separate CEMADEN contract test (`@integration`) | Undocumented API may change schema without notice; daily test detects before production breaks |
| `conftest.py` with `db_temporario` | Centralizes temp database setup — eliminates duplicate monkeypatch in 3 tests |
| Daily schedule for contract (not on push) | Avoids burning CI minutes on every commit for a test that depends on external network |
| Roadmap renumbering (4-7 → 4-8 with new Camada 5) | Event correlation is its own responsibility, not a sub-task of Camada 4. Document planning errors explicitly instead of hiding via "Camada 4.5" |
| Surrogate key + composite UNIQUE instead of composite PK | `id INTEGER PRIMARY KEY AUTOINCREMENT` + `UNIQUE (fonte, cod_alerta)`. FKs in other tables become simple INTEGERs; future URLs stay opaque (`/alertas/12345`); renaming sources does not break references |
| `cod_alerta` as TEXT, not INTEGER | CEMADEN uses numeric codes, EONET uses strings (`EONET_5421`). String is the most inclusive type |
| `municipio` optional, `coordenadas` required on Alerta | Inversion of previous rule. Multiple sources showed the minimum guaranteed across the set is coordinates, not municipality. Data honesty applied to the set, not CEMADEN alone |
| `NivelRisco.INDETERMINADO` added to enum | EONET does not classify severity. Type remains non-null; downstream treats `INDETERMINADO` as explicit category instead of special None case |
| `EscopoGeografico` enum instead of bool | Three states (BRASIL, PROXIMO, INTERNACIONAL) capture geographic relevance without inventing precision. Border and nearby maritime events have their own classification |
| Bbox + buffer instead of shapely polygon | Four numeric comparisons, no new dependency. "Data honesty" — imprecision is assumed and documented, replaceable by shapely without breaking contract |
| Geographic bands configurable in value, immutable in structure | Individual buffers via env var (`ALERTAVIDA_BUFFER_PROXIMO_GRAUS`). Adding/removing categories requires code change in `FAIXAS_DEFAULT` + migration via `scripts/reclassificar_escopos.py` |
| `escopo_geografico` pre-computed at ingestion | Computation at ingestion avoids recalculation in queries. Buffer changes only affect new alerts; re-classification is manual via dedicated script |
| Global NASA EONET ingestion (no `bbox` filter on request) | Brazil/Próximo/Internacional filtering happens in domain, not at source. Allows user to view events outside Brazil when desired (Camadas 7-8). Storage cost acceptable until Postgres migration |
| `monitor.py` becomes multi-source orchestrator (Camada 4) | Remains the entrypoint (`python -m alertavida.monitor`); CEMADEN-specific logic migrated to `sources/cemaden.py` in B.1.b (2026-05-16); loop over `[CemadenSource(), NasaEonetSource(), ...]` assembled by orchestrator in B.2 |
| Spatial indexing required in Camada 5 | SQLite R-Tree in current phase; PostGIS when migrating to Postgres in Camada 6. Not relevant for Camadas 1-4 |
| Refactor of `TipoEvento` enum to COBRADE subgroups | Domain depended on a single source's terminology (CEMADEN), violating Dependency Inversion. Refactor aligns with international COBRADE/EM-DAT standard. Each `DataSource` implements its own mapping. Low-cost window: done now, before Partes B/C triple the change surface |
| `cobrade_codigo` + `FonteClassificacao` enum on Alerta | `cobrade_codigo: str \| None` preserves subgroup-level code. `FonteClassificacao` enum (DIRETA, MAPEADA_POR_NOME, INFERIDA_POR_CONTEXTO, INDETERMINADA) registers classification provenance. Audit trail preserved when Camada 5 reclassifies to subtypes. Open/Closed: new classification strategies only add enum values |
| COBRADE granularity limited to subgroup in Camada 4 | Empirical inspection of 4 CEMADEN samples (240 alerts, 01-02/05/2026) confirmed no explicit COBRADE field and taxonomy limited to 2 types × 3 levels. Subtype distinction (flood vs flash flood vs waterlogging; falls vs landslides vs flows) requires topography, urban density, INMET. Heuristic inference in Camada 4 would violate data honesty (§6.10). Subtypes are Camada 5's concern |
| Subdivision of Camada 4 Parte A into A.1 (destructive) + A.2 (additive COBRADE) | A.1 breaks surrogate key, enum, domain refactor. A.2 is purely additive: nullable column, new module, no rewrites. Each part with coherent narrative and independently reversible. Order A.1 → A.2 avoids migration conflict. A.2 can run in parallel with Parte B |
| Versioned CEMADEN fixtures in `tests/fixtures/cemaden/` | 4 samples captured on 01-02/05/2026 (during real flooding in PE) become regression test fixtures. Usage by test level: parsing uses 1 fixture, ChangeDetector integration uses 2 consecutive fixtures with controlled diff, real contract remains via `@integration` on live endpoint |
| `data/` is runtime/state, gitignored with `.gitkeep` exception | SQLite DB, raw payloads, inspector logs, backups — nothing canonical. Uniform strategy: `data/*` ignored, structure preserved via `.gitkeep`. Canonical fixtures live in `tests/fixtures/`, not `data/samples/`. Single Responsibility applied to folder structure |
| COBRADE atomicity invariant via Pydantic `@model_validator`, not SQL CHECK constraint | SQLite does not allow adding CHECK constraint via `ALTER TABLE`, only via table rebuild. Using CHECK would break the `_migrar_banco()` path on pre-existing databases (A.2 would cease to be purely additive). Moving invariant to domain (`@model_validator(mode="after")` on `Alerta`) covers new and migrated databases uniformly, generates explicit `ValidationError` at the boundary, and aligns with §6 ("Pydantic for all external data I/O. Validation at system boundary") |
| Explicit schema verification before `_migrar_banco()` | The C3 → A.1 migration never existed (composite PK → surrogate key is not ALTER TABLE-able in SQLite). `CREATE TABLE IF NOT EXISTS` silently masked the incompatibility, and post-A.2 `_migrar_banco()` would add COBRADE columns to C3 databases, creating C3+A.2 chimera where current code's queries would break at runtime far from the root cause. Verification aborts with clear error asking to recreate the database. Path 3 of the 2026-05-12 architectural discussion |
| `SchemaIncompativelError` as specific exception | Facilitates `pytest.raises(SchemaIncompativelError)` and filtering in future logs. Not generic `RuntimeError` — distinguishes schema error from any other database initialization failure |
| `fonte` as attribute of `Alerta` model (B.0) | Source provenance is domain metadata, consistent with `fonte_classificacao` (A.2) which registers COBRADE classification provenance. Conceptual inconsistency in keeping one in domain and the other as operation parameter. Attribute on model eliminates propagated parameter in `aplicar_resultado_deteccao`, guarantees origin immutability via Pydantic frozen, makes `Alerta` self-sufficient for auditing (REPL, logs, tests). Cost: `Alerta.from_dict()` gains `fonte` parameter |
| `DataSource` as ABC with typed `ResultadoColeta`, not `list[Alerta]` | Orchestrator needs `descartados` per source to preserve the sanity assertion (`novos + atualizados + inalterados + descartados + erros == total`) that currently lives globally. Bare `list[Alerta]` forces the orchestrator to guess how many were discarded. `ResultadoColeta` (frozen dataclass with `alertas`, `descartados`, `coletado_em`) carries explicit accounting |
| `FalhaDeColeta` as domain typed exception | Same philosophy as `SchemaIncompativelError`: specific exception facilitates `pytest.raises(FalhaDeColeta)` and log filtering. Not generic `RuntimeError` — distinguishes source failure from any other orchestrator failure. Individual alert failures within a round do NOT raise `FalhaDeColeta` (counted in `descartados`); only round-level failures (network exhausted, schema rejected, retries exhausted) |
| Orchestrator separated into `ingestion/orquestrador.py`, not in `monitor.py` (B.2) | Current `monitor.py` violates SRP three times: entrypoint + orchestration + CEMADEN logic. B.1 extracts CEMADEN logic to `sources/cemaden.py`; B.2 extracts orchestration to `ingestion/orquestrador.py`; `monitor.py` becomes pure entrypoint (~30 lines). Aligns with target structure documented in §4 (Camada 1 subpackage `ingestion/`) |
| Typed `RelatorioIngestao` instead of loose counters | Today's "sanity assertion" lives in fragile local variables (`novos`, `atualizados`, `inalterados`, `descartados`, `erros`). `RelatorioFonte` (frozen dataclass per source) + `RelatorioIngestao` (frozen aggregate with `@property total`) makes accounting immutable, typed, and consumable by: logging, tests, and (future) `/health` endpoint of Camada 6. Invariants verifiable instead of local variables |
| Parametrized contract tests in `tests/sources/contrato.py` | Suite `verificar_contrato_data_source(source_factory)` runs on any `DataSource`. When `NasaEonetSource` comes (Parte C), one line of test guarantees full contract compliance. Contract changes (new field in `DataSource`, new invariant in `ResultadoColeta`) break ALL sources in a single definition. Centralized maintenance, safe evolution |
| `FakeDataSource` instead of `unittest.mock.Mock` | `Mock` is generic — any accessed attribute returns another Mock, hiding signature errors. `FakeDataSource` implements the real interface; if `DataSource` changes (new method, new signature), the fake breaks immediately giving feedback. Principle: real tests use doubles faithful to the real contract |
| Migration of Parte B in three chained commits (B.0 → B.1 → B.2), not big-bang | Same principle as A.1.1 → A.1.4: localized review, independent reversibility, CI green per step. B.0 changes domain without touching sources; B.1 extracts without reorganizing; B.2 reorganizes with the interface already established. Big-bang would concentrate risk in one giant commit where failures become treasure hunt |
| `FonteDado` as `StrEnum` (not `str, Enum`) | Consistency with `FonteClassificacao` (already `StrEnum`). `StrEnum` (Python 3.11+) returns value directly in `str(member)`, desirable for JSON serialization in event payloads. Decision does NOT extend retroactively to `NivelRisco`, `TipoEvento`, `EscopoGeografico` — outside B.0 scope; leaves this debt documented for future consolidation refactor |
| `FonteDado.from_string` strict (raises), does not return sentinel | Aligned with `NivelRisco.from_string`, not with `TipoEvento.from_string` which returns `INDETERMINADO`. Rationale: unknown source at runtime is a serious bug — database would write invalid value violating `UNIQUE(fonte, cod_alerta)`. Raising forces treatment. Source is always known at collection time (`DataSource` declares its origin), unlike `TipoEvento` where raw payload fields may be unrecognizable |
| Surgical `Annotated[FonteDado, Strict()]` instead of global `strict=True` in `model_config` | `strict=True` in `ConfigDict` would affect ALL `Alerta` fields, blocking coercion that currently works (ISO strings to datetimes, strings to `TipoEvento`/`NivelRisco` in direct construction). Non-surgical change would break an unpredictable number of existing tests. `Annotated[FonteDado, Strict()]` applies strict ONLY to the `fonte` field — blocks string → enum coercion where we want it, preserves rest of model. Pydantic v2 feature |
| `Alerta.from_dict(data, *, fonte: FonteDado)` keyword-only required | Eliminates ambiguity at the call site: `Alerta.from_dict(payload, FonteDado.CEMADEN)` would be less readable than `Alerta.from_dict(payload, fonte=FonteDado.CEMADEN)`. Pattern recommended in Effective Python item 25. Cost: zero. Benefit: clarity |
| `fonte_por_codigo` on `ResultadoDeteccao` instead of parameter to `aplicar_resultado_deteccao` | Decision revised 2026-05-13 during critical analysis before B.0.b. Rejected alternative (snapshots as parameter of `aplicar_resultado_deteccao`) would violate Dependency Inversion — Infrastructure (`database.py`) depending on Domain details (`AlertaSnapshot`) to discover source. Tell, Don't Ask: the detector "tells" everything that happened; infra "executes" without needing to guess. Detector already has access to `Alerta.fonte` and `AlertaSnapshot.fonte` — populates the map with no extra work |
| Retroactive domain modification acceptable in B.0.b | B.0.a closed domain layer. B.0.b needed to add `fonte_por_codigo` to `ResultadoDeteccao` — retroactive modification. Acceptable because: (1) B.0.a code still works, only gains a new field; (2) the alternative (redoing B.0.a with correct decision from the start) would waste already-committed work; (3) history honestly shows "B.0.a closed domain with incomplete decision, critical analysis before B.0.b refined". Principle: refactoring is legitimate work; hiding it via squash loses information |
| `FonteDado.from_string` on database read as safety net | `buscar_snapshots` calls `FonteDado.from_string(row[1])` when constructing `AlertaSnapshot`. If the `fonte` column ever has an invalid value (corruption, failed migration, wrong manual intervention), it raises `ValueError` on read instead of propagating silently to the domain. Cost: two extra characters. Benefit: early data corruption detection |
| `RelatorioFonte` with invariants via `__post_init__`, not external validation | Frozen dataclass with `__post_init__` validating sum `novos + atualizados + inalterados + descartados == coletados` (when `falha_coleta=False`), all counters zero and `coletado_em=None` (when `falha_coleta=True`), and `duracao_segundos >= 0`. Invariants live in the type, not in fragile local variables of the orchestrator. Invalid construction raises `ValueError` immediately |
| `agora_da_rodada` generated once by orchestrator, propagated to all sources | `executar_ingestao(agora: datetime \| None = None)` generates `datetime.now(UTC)` at start if not provided and propagates to all `aplicar_resultado_deteccao` calls. Guarantees two sources collected in the same round have the same timestamp in `visto_ultima_vez`, maintaining detector's `rodadas_ausente` counter coherence. Each `ResultadoColeta.coletado_em` records **when that source responded** — separate observability metric, not to be confused with round watermark. "State boundary" pattern in multi-source pipelines |
| Silent orchestrator (zero `print`, zero `logging`) | `executar_ingestao` returns typed `RelatorioIngestao` and stays silent. Presentation is the caller's responsibility (`monitor.py` in B.2.b, future `/health` endpoint in Camada 6). Logging is also presentation — for sysadmins instead of users. Deciding "logging vs print" can wait for Camada 6 with dedicated observability infra. SRP: orchestrator produces data, consumers present |
| `TipoEventoDetectado` StrEnum in detector, NOT a refactor of other domain enums | StrEnum (Python 3.11+) represents closed universe of event types produced by `ChangeDetector` — `CRIADO`, `ATUALIZADO`, `RESOLVIDO`. Type safety at check time (mypy `--strict` catches typo of member), exhaustive match/case possible, IDE autocomplete. `StrEnum.value` matches previous string — INSERT/SELECT in database continue storing `"AlertaCriado"` etc., zero data migration. Decision does NOT extend retroactively to `NivelRisco`, `TipoEvento`, `EscopoGeografico`, `FonteClassificacao` |
| `events.py` keeps raw strings in `bus.subscribe`, does NOT import `TipoEventoDetectado` | Decision revised 2026-05-17 during hardening-1. Initial analysis promoted StrEnum to `events.py`. Critical analysis before commit revealed that `EventBus` operates on raw `dict` — before hardening-1, `events.py` had no domain dependency. Importing `TipoEventoDetectado` would create Infrastructure → Domain dependency that did not exist. Dependency Inversion violation. `database.py` keeps it because it already receives `EventoDetectado` as parameter (extends existing dependency). Principle: refining existing dependency is legitimate; creating new dependency for cosmetic type safety is not |
| `AlertaResolvido` verified via outbox SQL, NOT via field on `RelatorioFonte` | `RelatorioFonte` represents what happened **from the collected batch** — resolved alerts did not come in the batch (by definition, they are absent). Adding `resolvidos: int` would mix dimensions. Resolution is a phenomenon **derived** from absence over N rounds, observed by `ChangeDetector` crossing active snapshots with current batch. Correct counter location is a dedicated observability endpoint in Camada 6 or Prometheus metric |
| `criar_banco()` called ONCE at scheduler startup, NOT every round | Schema setup belongs to bootstrap, not execution cycle. `agendar_ingestao()` calls `criar_banco()` before `scheduler.start()`, guaranteeing schema errors (incompatibility, migration needed, `SchemaIncompativelError`) happen at startup — operator sees the error immediately, not at 3am when the first tick runs. Calling every `_rodar_rodada` would be unnecessary I/O (~105k redundant calls/year). Industry standard: Django runs `migrate` before `runserver`, SQLAlchemy runs `Base.metadata.create_all()` at startup, FastAPI lifespan events do pre-loop setup |
| `monitor.py` reduced to pure entrypoint, orchestration logic centralized in scheduler | Pre-B.2.b, `monitor.py` violated SRP three times: it was entrypoint, contained orchestration logic (`executar_ingestao`), and had dead code from previous version. Post-B.2.b: `monitor.py` is 46 lines with single responsibility — debug/manual development via `python -m alertavida.monitor` executing ONE round. Production orchestrates via `agendar_ingestao()` in `scheduler.py`, which imports `executar_ingestao` from canonical namespace `alertavida.ingestion.orquestrador`. Both call `criar_banco()` at setup and `executar_ingestao([CemadenSource()])` in the ingestion flow |
| `main()` as real function in `monitor.py`, no alias `main = executar_ingestao` | The old alias served to make `python -m alertavida.monitor` run the orchestration function. In B.2.b, `main()` became a real function with entrypoint responsibilities: calling `criar_banco()`, instantiating source, calling orchestrator, formatting/printing report, returning exit code 0. Idiomatic Python: entrypoints return `int`, called via `raise SystemExit(main())`, separate I/O logic. `_formatar_relatorio` extracted as pure function for unit testability |
| `NasaEonetSource` builds `Alerta` directly, not via `from_dict` (C.1) | `Alerta.from_dict` is shaped for the CEMADEN payload (flat `latitude`/`longitude`, a `nivel` field, Portuguese `tipoevento`). The EONET v3 payload diverges on every axis: nested GeoJSON `geometry[].coordinates` `[lon, lat]`, no severity at all, English `categories[].id`, date per geometry fix. Reusing `from_dict` would discard every EONET event (`NivelRisco.from_string(None)` raises). Direct construction in the source matches the documented doctrine "each `DataSource` implements its own mapping" (invariant 10) — `CemadenSource` reuses `from_dict` only because its shape happens to match |
| EONET `nivel_risco = INDETERMINADO`, never inferred from magnitude (C.1) | EONET provides `magnitudeValue`/`magnitudeUnit` (acres, hectares, kts) but no severity. Deriving a risk level from incomparable magnitude units would be heuristic invention violating data honesty. `NivelRisco.INDETERMINADO` (added pre-implementation) is the honest mapping; downstream treats it as an explicit category |
| EONET source uses the most-recent geometry fix by date, not list order (C.1) | An EONET event has 1..N geometry fixes (median 1, max 52 in empirical inspection). `_fix_mais_recente` selects the fix with the greatest `date` as the alert's current position/time — the API does not guarantee chronological ordering of fixes. Single-fix events (the vast majority) are unaffected |
| EONET numeric COBRADE deferred to C.2; C.1 ships `cobrade_codigo=None` | The wiki's original C.1 bullet folded COBRADE mapping into C.1. Assigning numeric COBRADE codes (1.x.y.0.0) requires matching the official Defesa Civil table; doing it under uncertainty would violate "não inventar mapeamentos baseado em suposição". C.1 ships category→`TipoEvento` (unambiguous group level) with `cobrade_codigo=None` / `INDETERMINADA` (atomic invariant respected); the numeric mapping is C.2's focused, researched commit |
| EONET production query is `status=open` | Active/open events only (matches the empirical inspection's Requisição A). `status=all&days=30` is mostly closed historical events — noise for a real-time alert system. `url` is injectable for alternative environments |
